# Simplification Patterns

Use this reference to choose a format that makes a hard idea easier without making it false.

## Audience Levels

- **Total beginner:** remove most jargon, use one everyday analogy, explain why the topic exists.
- **Smart but new:** keep key terms, define them once, show how the parts connect.
- **Busy decision-maker:** lead with consequence, tradeoffs, and what action is implied.
- **Learner trying to remember:** use a sticky comparison, a sequence, and a tiny recap.

## Explanation Moves

- **Name the job:** "This exists to..."
- **Separate the parts:** "There are three pieces..."
- **Show the flow:** "First this happens, then this, then this."
- **Use a small example:** choose numbers, people, files, orders, or objects the user can picture.
- **Compare against the familiar:** "It is like X, except..."
- **Contrast with what it is not:** useful when two ideas are easily confused.
- **Explain the failure mode:** "The thing that goes wrong is..."
- **Give the rule of thumb:** include only when it is reliable enough to be useful.

## Good Simple References

Choose references that match the shape of the problem:

- **Gate / bouncer:** permissions, authentication, approval, filtering.
- **Recipe:** ordered steps, dependencies, repeatable process.
- **Map:** navigation, architecture, where pieces live.
- **Library:** search, indexing, categorization, retrieval.
- **Assembly line:** pipelines, staged processing, queues.
- **Bank account:** balances, credits/debits, budgets, limits.
- **Translator:** conversion between formats, protocols, schemas, or audiences.
- **Checklist:** validation, requirements, acceptance criteria.
- **Conversation:** request/response systems, APIs, negotiation, feedback loops.

Avoid analogies that add more concepts than they remove. If an analogy needs a long explanation, skip it.

## Plain-Language Checks

Before answering, check:

- Can the main idea fit in one sentence?
- Did every paragraph do a different job?
- Did any jargon survive without a definition?
- Did the example use concrete nouns?
- Did simplification hide an important warning, exception, or tradeoff?
- Could a beginner repeat the gist after reading it once?

## Templates

### Dense Concept

1. "In plain English: ..."
2. "The pieces are..."
3. "A simple way to picture it is..."
4. "The part people usually miss is..."
5. "What to remember: ..."

### Confusing Text

1. "This is saying..."
2. "It matters because..."
3. "The plain-English version is..."
4. "Questions to ask next..."

### Technical Problem

1. "The system is trying to..."
2. "The problem is..."
3. "The likely cause is..."
4. "A simple reference: ..."
5. "The next practical step is..."

### Code

1. "This code's job is..."
2. "Data comes in here..."
3. "It changes here..."
4. "It leaves here..."
5. "The risky or tricky part is..."
